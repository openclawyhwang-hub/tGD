COMMAND 1 / 7

/tgd-map — Map

Scan and understand the existing project context before making changes.

⚙️ Triggers

context-engineering
codegraph init

🔒 Pre-flight

None — this is the entry point. Can run on any project.

📋 Process

1. Analyze tech stack, architecture, dependencies
2. Map code organization and existing patterns
3. Initialize CodeGraph (offline knowledge graph)
4. Create symlinks for graph DB storage

📦 Output

Output Structure
$TGD_DIR/                   ← Sibling to project repos
├── CONTEXT.md             ← Product-level context
│   - Tech stack
│   - Architecture
│   - Repo inventory
│   - Dependencies
└── .scans/                ← Scan data (centralized)
    └── <repo>/
        ├── .codegraph/    ← Knowledge graph DB
        └── .understand-anything/

💡 Usage

Terminal
$ /tgd-map
Agent: Scanning project...
→ Tech stack: Next.js 15, TypeScript, Prisma
→ 47 source files, 12 test files
→ Architecture: App Router + API routes
→ $TGD_DIR/CONTEXT.md created ✓
MAP — Context before changes

COMMAND 2 / 7

/tgd-define — Define

Extract intent, name the feature, write PRD + SPEC before any code.

⚙️ Triggers

interview-me
idea-refine
spec-driven-development

🔒 Pre-flight

Requires $TGD_DIR/CONTEXT.md — run /tgd-map first.

📋 Process

1. 3-Option Naming — propose 3 kebab-case names, user picks one
2. Git Branch — create feature/<name> from main
3. Interview — extract intent if underspecified
4. Idea Refine — stress-test if concept is vague
5. PRD + SPEC — write structured requirements
6. UI Gate — ask if DESIGN.md is needed

📦 Output

Output Structure
$TGD_DIR/
└── <feature-name>/

    │   - Objectives
    │   - User stories
    │   - Scope & boundaries
    ├── SPEC.md        ← Technical Spec
    │   - API contracts
    │   - Data models
    │   - Error handling
    │   - Test strategy
    ├── DESIGN.md      ← (optional) UI Design
    │   - Component tree
    │   - Visual specs
    │   - Responsive rules
    └── prototype/     ← (optional) HTML mockups
        ├── variant-a.html
        └── variant-b.html

COMMAND 3 / 7

/tgd-plan — Plan

Decompose specs into small, verifiable tasks with acceptance criteria.

⚙️ Triggers

planning-and-task-breakdown
jira-auto-sync (conditional)

🔒 Pre-flight

✓ $TGD_DIR/CONTEXT.md exists
✓ $TGD_DIR/<feature>/PRD.md exists
✓ $TGD_DIR/<feature>/SPEC.md exists

📋 Process

1. Read CONTEXT.md — project structure
2. Read PRD.md — business goals
3. Read SPEC.md — technical requirements
4. Decompose into atomic, testable tasks
5. Add acceptance criteria per task
6. Optional: sync to Jira

📦 Output

Output Structure
$TGD_DIR/
└── <feature-name>/
    └── TASKS.md       ← Task breakdown (BDD)
        ## Task 1: Setup auth middleware
        - Given valid JWT → When API call → Then 200  [R]
        - Files: src/middleware/auth.ts
        - Tests: tests/auth.test.ts

        ## Task 2: Login endpoint
        - Given valid creds → When POST /login → Then 200 + token  [R]
        - Given wrong pwd → When POST /login → Then 401  [R]
        - Files: src/routes/auth.ts
        - Tests: tests/login.test.ts

[R] = regression marker (acceptance-level test)

Sign-off:
  - [ ] DEV: (pending)
PLAN — Triple-source task breakdown

COMMAND 5 / 7

/tgd-verify — Verify

Hard Gate. Tests are proof. No exceptions.

⚙️ Triggers (Hard Gate)

debugging-and-error-recovery
test-driven-development
agent-browser (mandatory)

🔒 Pre-flight

✓ Source code exists in src/
✓ Test files exist in tests/

📋 Process

1. Run ALL tests (unit + integration + E2E)
2. Debug any failures: reproduce → localize → fix → guard
3. Launch agent-browser for UI E2E verification
4. Capture screenshot as visual evidence
5. BLOCK if anything fails

📦 Output

Output Structure

    └── TEST-REPORT.md    ← Auto-generated test results

        [R] test_valid_login       ✅
        [R] test_wrong_password    ✅
        test_empty_password        ✅
        ...

Code Repo: tests/ (all passing)

Sign-off:

COMMAND 6 / 7

/tgd-review — Review

Five-axis code quality review before merge.

⚙️ Triggers

code-review-and-quality
code-simplification
security-and-hardening
performance-optimization

🔒 Pre-flight

✓ Test files exist in tests/
✓ All tests pass (verified by /tgd-verify)

📋 Five Axes

1. Correctness — Does it do what SPEC says?
2. Maintainability — Can someone else understand it?
3. Security — OWASP + secrets management
4. Performance — No N+1, no unnecessary re-renders
5. Test Coverage — 80% unit, 15% integration, 5% E2E

📦 Output

Output Structure

└── <feature-name>/
    └── REVIEW.md          ← Review report

        ✅ Performance: No issues
        ✅ Coverage: 87%

        ## Issues Found
        - [MEDIUM] /api/login: no rate limit
        - [LOW] user.ts: unused import

        ## Recommendation
        Fix rate limit, then ready to ship.

TESTING STRATEGY

Three roles. Three stages.
One rule: no shortcuts.

🔨 Builder

/tgd-develop

Write tests alongside code. Red → Green → Refactor.

🔍 Inspector

/tgd-verify

Run ALL tests. Reproduce → Localize → Fix → Guard.

🕵️ Auditor

/tgd-review

Audit test quality. 80% unit, 15% integration, 5% E2E.

Why three stages? Separation prevents "lazy agent" behavior.
