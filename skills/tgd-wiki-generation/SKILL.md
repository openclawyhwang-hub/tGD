# tGD Wiki Generation (Docusaurus, Multi-Repo)

## Overview

Compile the outputs of CodeGraph (`.codegraph/codegraph.db`) and
Understand-Anything (`.understand-anything/knowledge-graph.json`) for
**every** repo scanned in `$TGD_DIR/.scans/` into a single Docusaurus 3
documentation site under `$TGD_DIR/wiki/`.

Each repo lands in its own subtree at `docs/repos/<slug>/` with a full
wiki (overview, architecture, onboarding, modules, flows, diagrams).
The **primary repo** (first scan or `--primary <slug>`) is highlighted
with ⭐ in the top-level home page and navbar dropdown.

**Design guarantee:** every repo generated by this skill lands on the
same layout — same brand colors, same Grid Cards, same React components —
because all styling and layout assets ship inside the skill and are copied
into `$TGD_DIR/wiki/src/` on every run. Users never edit these; they are
overwritten on every regeneration.

## When to Use

- Called automatically by `/tgd-map` Step 6 after CodeGraph + UA analysis complete
- Called manually to regenerate the wiki after code changes
- Called via `/tgd-wiki` slash command (future addition)

## Inputs

Required:

- `$TGD_DIR` — resolved by `/tgd-map` Step 0
- `$TGD_DIR/.scans/<repo>/.understand-anything/knowledge-graph.json` for each repo
- Optional: `$TGD_DIR/.scans/<repo>/.codegraph/` (used if `codegraph` CLI is available)

The primary repo is the first entry in the scans directory unless `--repo` is
passed. Additional repos (added in Step 1 of `/tgd-map`) are treated as
context repositories and get their own module pages.

## Outputs

All wiki artefacts live under `$TGD_DIR/wiki/` — one self-contained folder.
Every scanned repo gets its own full wiki tree under `docs/repos/<slug>/`;
`docs/index.mdx` is the shared home with a repo grid + navbar dropdown.

```
$TGD_DIR/
├── CONTEXT.md                    ← tGD core (untouched)
├── .scans/                       ← tGD core (untouched)
└── wiki/                         ← ALL wiki artefacts live here
    ├── docs/
    │   ├── index.mdx             ← top-level home (repo grid)
    │   ├── sources.mdx           ← all-repos summary page
    │   ├── manifest.json         ← top-level manifest (all repos)
    │   └── repos/
    │       ├── <primary-slug>/   ← ⭐ primary repo, full wiki tree
    │       │   ├── index.mdx     ← repo home (hero + module grid)
    │       │   ├── overview.mdx
    │       │   ├── architecture.mdx
    │       │   ├── onboarding.mdx
    │       │   ├── modules/*.mdx ← one per architectural layer
    │       │   ├── flows/*.mdx   ← one per tour step
    │       │   ├── source/*.mdx  ← offline source browser + line anchors
    │       │   ├── diagrams/
    │       │   │   ├── index.mdx
    │       │   │   ├── architecture.mmd
    │       │   │   └── dependencies.mmd
    │       │   └── manifest.json ← per-repo manifest
    │       └── <other-slug>/     ← each additional context repo,
    │           └── ... same tree ...   with the same wiki structure
    ├── src/
    │   ├── components/           ← React components (copied from skill on every run)
    │   │   ├── ModuleCard.tsx
    │   │   ├── KPIGrid.tsx
    │   │   ├── LayerBadge.tsx
    │   │   └── Hero.tsx
    │   └── css/
    │       └── custom.css        ← tGD brand theme (copied from skill on every run)
    ├── docusaurus.config.ts      ← auto-generated (with Repos dropdown when > 1 repo)
    ├── sidebars.ts               ← auto-generated (one category per repo)
    ├── package.json              ← auto-generated with pinned deps
    ├── .gitignore                ← ignores node_modules/ + build/
    ├── node_modules/             ← npm install output (gitignored)
    └── build/                    ← docusaurus build output
        └── index.html
```

Three coordinated entry points:

| Audience | Entry point | Purpose |
|---|---|---|
| Human | `$TGD_DIR/wiki/build/index.html` (or serve via `npm run start`) | Browse the wiki |
| Agent | `$TGD_DIR/wiki/docs/manifest.json` | Top-level index of every scanned repo |
| Agent (per-repo) | `$TGD_DIR/wiki/docs/repos/<slug>/manifest.json` | Deep dive into one repo |
| tGD stages | `$TGD_DIR/CONTEXT.md` | Top-level summary (unchanged) |

## Execution

```bash
python <SKILL_DIR>/scripts/generate-wiki.py "$TGD_DIR"
python <SKILL_DIR>/scripts/generate-docusaurus-config.py "$TGD_DIR"
bash   <SKILL_DIR>/scripts/build-site.sh "$TGD_DIR"
```

`<SKILL_DIR>` resolves to the directory containing this SKILL.md.

Each script fails soft:

- `generate-wiki.py` — hard failure aborts (docs are required)
- `generate-docusaurus-config.py` — hard failure aborts
- `build-site.sh` — soft failure logs a warning and continues
  (MDX remains readable when `npm` is missing or `npm install` fails)

## Dependencies

| Tool | Purpose | Required? | Install |
|---|---|---|---|
| Python 3.9+ | Run generation scripts | Required | System |
| Jinja2 | Template rendering | Required | `pip install jinja2` |
| Node.js 22+ | Docusaurus runtime | Optional (for build) | `nvm install 22` / package manager |
| npm 10+ | Package management | Optional (for build) | Bundled with Node |
| codegraph CLI | Symbol-level pages | Optional | See CodeGraph docs |

Optional dependencies:

- When Node/npm is missing, the skill still produces `docs/*.mdx` and
  `manifest.json`. Only `npm install` + `docusaurus build` are skipped.
- The static site build takes ~30 seconds on first run (npm install pulls
  ~200MB); subsequent builds are ~5-10 seconds.

## Version Pinning

Docusaurus dependencies are pinned via `package.json` `^` ranges + a
committed `package-lock.json` template (see `assets/package-lock.json`).
This gives corporate networks maximum flexibility (allows patch/minor
upgrades) while keeping every fresh install byte-identical thanks to the
lockfile.

Pinned versions (as of skill v0.2.0):

| Package | Version |
|---|---|
| @docusaurus/core | ^3.7.0 |
| @docusaurus/preset-classic | ^3.7.0 |
| @docusaurus/theme-mermaid | ^3.7.0 |
| @mdx-js/react | ^3.0.0 |
| react | ^18.2.0 |
| react-dom | ^18.2.0 |

## Regeneration

Re-running the skill on the same `$TGD_DIR` overwrites `docs/`,
`docusaurus.config.ts`, `sidebars.ts`, `package.json`, `src/components/`,
and `src/css/` in place. `manifest.json` is regenerated fresh — do not
hand-edit; edits will be lost.

**Guarantee:** the visual layout, brand colors, and component set are
uniform across all projects because they are shipped inside the skill
(`assets/css/`, `assets/components/`) and copied into `$TGD_DIR/wiki/src/` on
every run. Do not modify `$TGD_DIR/wiki/src/` files by hand — they will be
overwritten. If you need customization, patch the skill's `assets/`
directory instead.

## Pitfalls

- ❌ **Do not write into the code repo.** All outputs go under `$TGD_DIR/`.
- ❌ **Do not hand-edit `manifest.json`** — regenerated on every run.
- ❌ **Do not hand-edit `$TGD_DIR/wiki/src/`** — copied fresh from skill assets on every run.
- ❌ **Skipping `generate-wiki.py` fallback logic** — non-code files (config, docs) must still appear as wiki pages.
- ❌ **Not escaping MDX special chars** — `<`, `>`, `{`, `}` in generated content break MDX parsing. `generate-wiki.py` MUST escape.
- ❌ **Assuming `npm` is installed** — the site build must degrade gracefully.
- ❌ **Committing `node_modules/`** — must be in `.gitignore`.

## Common Rationalizations

| Rationalization | Reality |
|---|---|
| "Docusaurus is heavy — just use plain HTML." | Docusaurus gives React components, versioning, i18n, search, dark mode, Mermaid, MDX — rolling our own would take months and produce worse output. |
| "Users should be able to customize the theme." | Uniform layout is the design guarantee. Users who want custom theme fork the skill. |
| "MDX escape is overkill for simple summaries." | LLM-generated summaries routinely contain `<Component>` or `{expression}` snippets that break MDX. Escape is mandatory. |
| "npm install is fine at build time." | On corporate networks with strict proxies, `npm install` can fail. That's why the wiki produces MDX first and only builds if npm succeeds. |

## Red Flags

- Emitting MDX pages that reference nodes not present in the knowledge graph
- Writing anywhere outside `$TGD_DIR/` (code repo, `$HOME`, `/tmp` beyond scratch)
- Hard-failing when `npm` is missing instead of soft-failing on the build
- Generating `manifest.json` without listing every page written to disk
- Leaving stale files behind on regeneration
- ❌ **Not generating source pages** — symbol links must never point to external GitHub-only URLs; offline source pages under `docs/repos/<slug>/source/` are required.
- ❌ **Skipping MDX escape** (`<`, `{` in summaries/source snippets → build failure)
- Silently swallowing template rendering errors — every failure must surface
- Modifying `$TGD_DIR/wiki/src/components/` from anywhere other than skill assets
